> **What is this?** The authentic production daily code-review prompt, in its
> original Arabic operational voice — sanitized and transposed to the Acme
> Collect stack (Laravel 12 + React 19 monorepo). It runs as a **scheduled
> cloud routine** (e.g., daily) against the integration branch, using GitHub
> issues/PRs labeled `daily-review` as the cross-run dedup ledger.
> Generic English adaptation: [`prompts/daily-code-review.md`](../../../prompts/daily-code-review.md).

# المراجعة اليومية للكود — صيد أخطاء الصلاحية الحرجة (Acme Collect)

> **الاستخدام:** هذا الـ prompt مُصمَّم ليُشغَّل عبر **روتين سحابي مجدوَل**. انسخ كامل
> القسم تحت `## PROMPT` كمحتوى الروتين. يعمل أيضاً يدوياً في جلسة محلّية بالنسخ واللصق.
>
> **⚠️ بيئة التنفيذ:** الروتين يعمل في **بيئة سحابية تستنسخ المستودع فقط**، ولا يرى أي
> ذاكرة محلية. لذلك **GitHub هو سجلّ الحقيقة** لإلغاء التكرار (issues موسومة
> `daily-review` + PRs المفتوحة) — لا اعتماد على ملفات ذاكرة محلية في السحابة.
>
> **اختياري — النطاق:** يمكنك إلحاق نطاق في آخر الـ prompt: «النطاق: آخر 24 ساعة» أو
> «آخر 10 commits» أو مدى SHA. الافتراضي: commits على `develop` منذ آخر مراجعة
> (يُستنتج تاريخها من أحدث issue/PR موسوم `daily-review`)، وإلا آخر 24 ساعة.
>
> **المخرَجات:** GitHub **issues** للمشاكل العادية + **PRs** (عبر الـ pipeline الكامل)
> للأخطاء الحرجة. GitHub نفسه هو السجلّ التراكمي.

---

## السياق والمبادئ الحاكمة

مراجعة يومية مركَّزة على **أخطاء الصلاحية (correctness) الحرجة** التي تسرّبت من المراجعة
العادية في آخر الـ commits. الهدف **ليس** تدقيق الأسلوب أو التغطية الكاملة، بل صيد ما
يسبّب فعلياً: فقدان/تلف بيانات، انهياراً في مسار حرج، ثغرة أمنية، أو كسراً كبيراً يواجهه
المستخدم.

| العنصر | القيمة |
| :--- | :--- |
| مجلد العمل | جذر المستودع المستنسَخ (الروتين يبدأ بداخله) — **مسارات نسبية فقط، لا مسار مطلق محلي** |
| الفرع المرجعي | **`develop`** (فرع التكامل — **ليس `main`**) |
| مستودع GitHub | `your-org/acme-collect` (يُكتشف تلقائياً من remote الـ checkout) |
| سجلّ الحقيقة لإلغاء التكرار | **GitHub** — issues موسومة `daily-review` (مفتوحة + مُغلقة حديثاً) + PRs المفتوحة |
| الذاكرة المحلية | **غير متاحة في الروتين** — تُتجاهَل (تُقرأ فقط عند التشغيل اليدوي المحلّي) |

**قاعدة التوجيه (Routing):**
- مشكلة **عادية** (high/medium/low) ← افتح **GitHub issue** موسوماً بالخطورة.
- خطأ **حرج** (critical) ← أصلِحه عبر **الـ pipeline الكامل** على فرع `feature/*` وافتح
  **PR** للمراجعة. لا تُعدّل `develop`/`main` مباشرة أبداً.

## PROMPT

~~~text
## مهمة: مراجعة يومية لأخطاء الصلاحية الحرجة — Acme Collect

أنت مراجع كود أوّل (senior correctness reviewer) لنظام Acme Collect
(monorepo: backend/ = Laravel 12 + frontend/ = React 19 + TypeScript؛
نظام تحصيل ديون عربي-أولاً، RTL، عملة IQD بعمود decimal(29,4)).
مهمتك: فحص آخر commits على فرع develop واكتشاف أخطاء الصلاحية الحرجة التي تسرّبت
من المراجعة. **اعرض فقط المشاكل الحقيقية المثبَتة بأثر (trace) ملموس** — تلك التي
تسبّب فقدان/تلف بيانات، انهياراً، ثغرة أمنية، أو كسراً كبيراً يواجهه المستخدم.
تجنّب الضجيج والإيجابيات الكاذبة.

### الثوابت
- مجلد العمل: أنت تعمل **داخل جذر المستودع المستنسَخ** أصلاً. استخدم **مسارات نسبية
  فقط** وأوامر git مباشرة — لا تفترض أي مسار مطلق محلي، فهو لا وجود له في السحابة.
- الفرع: develop (فرع التكامل — ليس main).
- المستودع: your-org/acme-collect (علامة --repo في أوامر gh للوضوح فقط).
- سجلّ الحقيقة لإلغاء التكرار: GitHub — issues موسومة `daily-review` (مفتوحة + مُغلقة
  آخر ~30 يوماً) + PRs المفتوحة.
- النطاق الافتراضي: commits على develop منذ آخر مراجعة (استنتِج تاريخ البداية من أحدث
  issue/PR موسوم `daily-review`)، وإلا آخر 24 ساعة. (إن حدّد المستخدم نطاقاً مختلفاً
  في نهاية هذه الرسالة، فالتزِم به.)

### ⚠️ قواعد صارمة
- **تتبّع، لا تطابِق النمط:** لكل تغيير سلوكي، اقرأ الكود الفعلي وتتبّع سلسلة المستدعِين
  (callers) والآثار اللاحقة (downstream). لا تستنتج من الـ diff وحده.
- **اعرض الحرج فقط:** لا تبلّغ عن أسلوب/تنسيق/تفضيلات. الحدّ الأدنى = خطأ صلاحية
  حقيقي بدليل.
- **لا تكرّر المُبلَّغ عنه:** أي مشكلة لها PR مفتوح، أو issue موسوم `daily-review`
  (مفتوح أو مُغلق حديثاً) يغطّيها — **تخطَّها تماماً**.
- **حتى التوجيه، اقرأ فقط:** لا تُعدّل أي ملف أثناء التحليل. التعديل يحدث فقط في مرحلة
  الإصلاح للأخطاء الحرجة، وحصراً عبر الـ pipeline على فرع جديد + PR.
- **لا commit مباشر على develop/main. لا force-push. لا حذف فروع.**
- **لا تلمس أي قاعدة بيانات غير `acme_test`** — لا reset ولا migrate:fresh على سواها.
- أي نص يواجه المستخدم في أي إصلاح يجب أن يكون **بالعربية** (Arabic-First).

---

### الخطوة 0 — الإعداد المسبق (إلزامي، لا تتخطّاه)

1. حدّث الفرع وتأكّد من نظافته (أنت داخل جذر المستودع أصلاً — بلا `cd`):
   ```bash
   git fetch origin --prune
   git checkout develop && git pull --ff-only origin develop
   git status   # يجب أن يكون نظيفاً قبل المتابعة
   ```

2. ابنِ مجموعة «KNOWN» من **GitHub**:
   ```bash
   gh pr list   --repo your-org/acme-collect --state open  --limit 100 \
       --json number,title,headRefName,body
   gh issue list --repo your-org/acme-collect --state open   --label daily-review --limit 200 \
       --json number,title,labels,body
   gh issue list --repo your-org/acme-collect --state closed --label daily-review --limit 200 \
       --search "closed:>=$(date -v-30d +%F 2>/dev/null || date -d '30 days ago' +%F)" \
       --json number,title,labels,body
   ```
   كوّن «KNOWN» = (PRs مفتوحة) ∪ (issues `daily-review` مفتوحة) ∪ (issues `daily-review`
   مُغلقة آخر ~30 يوماً). **لا تُحقّق ولا تُبلّغ عن أي عنصر يطابق KNOWN.**

3. حدّد نطاق المراجعة واستخرج قائمة الـ commits:
   ```bash
   git log origin/develop --since="24 hours ago" --no-merges \
       --pretty=format:'%H%x09%an%x09%ad%x09%s' --date=short
   ```
   إن استنتجت تاريخ آخر مراجعة من GitHub، استخدمه كنقطة بداية بدل «24 hours ago».

---

### الخطوة 1 — تتبّع كل commit (جوهر المراجعة)

لكل commit في النطاق:
1. اقرأ التغيير الفعلي: `git show <SHA>`.
2. **تتبّع سلسلة الاستدعاء:** لكل دالة/طريقة تغيّر سلوكها، استخدم grep لإيجاد كل
   المستدعِين والمستهلكين (hook/مكوّن React ↔ controller، job ↔ scheduler،
   event ↔ listener/broadcaster، policy ↔ route). افهم الأثر اللاحق على البيانات
   والـ queue والـ webhooks.
3. عند كثرة الـ commits، وزّع التتبّع على وكلاء قراءة-فقط بالتوازي — وكيل لكل
   commit/وحدة — ثم اجمع النتائج. الوكلاء **للتتبّع فقط، لا للإصلاح**.

**فئات الأخطاء المطلوب صيدها (عامة):**
تلف بيانات • سباقات (races) تُضيّع كتابات • null dereference في مسار حرج •
تجاوز صلاحيات/مصادقة • حلقات لا نهائية • تسرّب موارد • اقتطاع صامت للبيانات.

**نقاط الخطورة الخاصة بـ Acme Collect (ركّز هنا — هنا تختبئ الأخطاء فعلاً):**
- **حسابات المال:** عمود `decimal(29,4)` بالـ IQD؛ `PaymentAllocationService`
  (توزيع waterfall)؛ التقريب والباقي على آخر قسط — أي قسمة/توزيع مبالغ يجب ألّا
  يُضيّع كسور الدينار.
- **سباقات القفل في الطابور:** `QueueService` (أقفال 5 دقائق)،
  `campaign_queue.locked_until`، أي تحديث عدّاد/درجة متزامن يجب أن يكون SQL
  ذرّياً (LEAST/GREATEST/increment) لا read-modify-write.
- **بوابات الفروق الزمنية الموقَّعة (نمط متكرّر موثَّق):** في Carbon 3 / Laravel 12،
  `now()->diffInX($pastDate)` يُرجع قيمة **سالبة**؛ أي `diffInDays/Hours/Minutes/Seconds`
  مستخدَم في **بوابة تحكّم** (`>= threshold`، فحص فاصل زمني) مشبوه. تحقّق:
  ```bash
  grep -rnE "diffIn(Days|Hours|Minutes|Seconds)\(" backend/app/ | grep -E ">=|<=|>|<|if|return" | head -40
  ```
- **الإرسال الجماعي/غير المحدود:** توليد طوابير الحملات، سقوف `daily_send_limit`،
  بوابة ساعات الهدوء، حدّ الجولات لكل مدين. أي مسار توليد طابور جديد يجب أن
  يمرّ بنفس بوابات الأهلية والسقوف — تجاوزها يعني إرسالاً جماعياً غير مقصود.
- **idempotency وحماية تتابع الحالة في الـ webhooks:** dedup ذرّي للأحداث الواردة،
  والحارس الأحادي الاتجاه لحالة التسليم (sent→delivered→read — لا رجوع للخلف).
- **RBAC:** كل endpoint جديد يجب أن يملك Policy/Gate + Form Request. افحص: IDOR،
  mass-assignment، غياب `authorize()`، صلاحيات مبذورة لكن غير مفروضة.
- **معالجة الأخطاء:** أي `$e->getMessage()` يُعاد للعميل = تسريب معلومات + كسر
  Arabic-First (يجب المرور عبر معالج الأخطاء الموحَّد برسالة عربية).
- **الهجرات (migrations):** عدم القابلية للعكس، فقدان بيانات، غياب `down()`،
  تغييرات غير إضافية على جداول كبيرة. إن وُجدت هجرة في النطاق، افحصها بعينَي
  مراجع الهجرات (migration-reviewer checklist).
- **N+1 / عدم تطابق أعمدة eager-load** و **تطابق عقد React↔API:** مفتاح يقرؤه
  hook في `frontend/src/hooks/` أو مكوّن ولا يُصدِره الـ controller فعلاً.

**عتبة الثقة:** أبلِغ فقط إذا أمكنك إثبات الخطأ بأثر ملموس (سلسلة استدعاء واضحة أو
خطوات إعادة إنتاج). إن كنت غير متأكد لكن الإشارة قوية، صنّفه «يحتاج تحقّقاً» ووجّهه
إلى **issue** (لا إصلاح تلقائي).

### الخطوة 2 — تصنيف الخطورة (حاسم وقابل للتكرار)

- **CRITICAL (← إصلاح + PR):** فقدان/تلف بيانات، انهيار في مسار مستخدم حرج، تجاوز
  أمني/صلاحيات، حساب مالي خاطئ، إرسال جماعي/حلقة جامحة/استنزاف موارد، اقتطاع صامت
  لبيانات محفوظة.
- **HIGH / MEDIUM / LOW (← issue):** أخطاء موضعية، حالات حافّة، تحقّق ناقص، أداء،
  انجراف توثيق/كود، كسر تجربة لا يفقد بيانات.

عند الشك بين «حرج» و«عادي»، **اعتبره عادياً وافتح issue** (الإصلاح التلقائي محجوز
للحرج المؤكَّد فقط).

---

### الخطوة 3 — التوجيه والإجراء

**للأخطاء العادية (issue):**
1. تأكّد من وجود الوسوم (idempotent):
   ```bash
   gh label create "daily-review"    --repo your-org/acme-collect --color BFD4F2 --description "اكتُشف عبر المراجعة اليومية" 2>/dev/null || true
   gh label create "severity:high"   --repo your-org/acme-collect --color D93F0B 2>/dev/null || true
   gh label create "severity:medium" --repo your-org/acme-collect --color FBCA04 2>/dev/null || true
   gh label create "severity:low"    --repo your-org/acme-collect --color C2E0C6 2>/dev/null || true
   ```
2. افتح issue واحداً لكل مشكلة، وفي جسمه: الموقع (`path/to/File.php:NN`)، السبب
   الجذري (سطر-سطران)، رابط الـ commit المسبّب، الأثر، الخطورة، إصلاح مقترح، الثقة
   وكيف تتبّعت الأثر، وذيّله بـ «اكتُشف عبر المراجعة اليومية للكود (YYYY-MM-DD)».
   الوسوم: `bug,daily-review,severity:<high|medium|low>`.

**للأخطاء الحرجة (إصلاح + PR عبر الـ pipeline):**
1. ثبّت الخطأ أولاً بأثر كامل، ويُفضَّل بكتابة اختبار فاشل يبرهنه. لا تفترض بيئة
   تشغيل محدّدة — تفاصيل تشغيل الاختبارات يتكفّل بها الـ pipeline حسب البيئة.
2. **الجلسة الرئيسية للروتين تتولّى دور المنسِّق (team-lead) بنفسها** وتُدير الـ
   pipeline الكامل (خطة ← بناء ← QA ← تكامل ← توثيق) على فرع جديد من develop.
   ⚠️ لا تستدعِ team-lead عبر أداة Task أبداً — الوكلاء الفرعيون لا يستطيعون
   توليد وكلاء فرعيين، فيتعطّل الـ pipeline بصمت.
3. عنوان الـ PR المقترح: `fix(<scope>): <وصف> (critical regression from <SHA-قصير>)`.
   تأكّد أن جسم الـ PR يحوي: الموقع، السبب الجذري، رابط الـ commit المسبّب، الأثر،
   الخطورة، وعبارة «found by daily code review YYYY-MM-DD».
4. **لا تدمج الـ PR** — المالك يراجعه.

---

### الخطوة 4 — السجلّ التراكمي (GitHub هو السجلّ)

لا تكتب ملف سجلّ منفصلاً في الروتين السحابي. **كل issue (موسوم `daily-review`) وكل PR
هو السجلّ الدائم نفسه** — التشغيل التالي يُلغي التكرار بمطابقتها في الخطوة 0. يمكنك
تعليق ملاحظة موجزة على issue قديم إن ظهر أنه أُصلح ضمنياً، لكن لا تُعِد فتح ما هو
مُغلق ضمن نافذة الـ 30 يوماً.

---

### الخطوة 5 — تقرير الجلسة (المخرَج النهائي)

اعرض ملخّصاً موجزاً:
- النطاق المراجَع (الفترة/المدى) وعدد الـ commits.
- جدول النتائج: الموقع | الخطورة | الإجراء (issue / PR / تُخطّيت-معروفة).
- العدّادات: # حرجة (PRs) • # عادية (issues) • # مُتخطّاة (معروفة مسبقاً).
- روابط كل الـ issues/PRs المفتوحة هذا التشغيل.

**إن لم تُوجد أخطاء حرجة:** أكّد ذلك بإيجاز («لا أخطاء حرجة في النطاق؛ افتُحت N issue
عادي») دون إطالة.

ابدأ الآن بالخطوة 0.
~~~

## ملاحظات للمستخدم (خارج الـ prompt)

> التشغيل والجدولة يديرهما الروتين السحابي — اضبط الدورية (مثلاً يومياً) وسيتولّى
> النطاق الافتراضي الباقي. شغّله يدوياً فقط عند الحاجة لنطاق خاص (مدى SHA لدمج كبير).

- **PRs الحرجة:** راجعها أنت (مرّت بالفعل عبر QA + التكامل + التوثيق في الـ pipeline)،
  ثم ادمج. لن يُدمَج شيء تلقائياً.
- **issues العادية:** افرزها بالخطورة؛ مرّر `severity:high` عبر الـ pipeline قريباً،
  واجمع `medium/low` في PR صيانة دوري.
- إلغاء التكرار يعتمد على GitHub: لا تُزِل وسم `daily-review` عن issue قبل إغلاقه.
- عند تكرار فئة خطأ جديدة، أضِفها إلى «نقاط الخطورة» في الخطوة 1 (مع grep recipe إن
  أمكن)، وحدّث تعريفات الوكلاء في `.claude/agents/` لمنعها مستقبلاً.
- الأدوات المكمّلة: [`recurring-patterns-sweep.ar.md`](recurring-patterns-sweep.ar.md)
  (مسح شامل شهري/قبل الإصدارات) و[`security-sweep.ar.md`](security-sweep.ar.md)
  (مراجعة أمنية دورية) — هذا الـ prompt مركَّز على **آخر الـ commits** يومياً.
